# Create Migration

This skill creates and applies an Alembic autogenerated migration for the stonks project. Run the full flow automatically without prompting — validate the result yourself and only stop if something looks genuinely wrong.

## Project context

- Models live in `backend/app/*/models.py`
- Migrations live in `backend/alembic/versions/`
- File naming: `YYYYMMDD_HHMM_slug_rev.py` (already configured in alembic.ini)
- Commands (run from repo root):
  - `just db-migrate <message>` — autogenerate a migration from model changes
  - `just db-upgrade` — apply all pending migrations
  - `just db-downgrade` — roll back by one revision

## Steps

### 1. Understand what changed

Check what model changes have been made:

```bash
git diff HEAD -- 'backend/app/*/models.py'
git diff -- 'backend/app/*/models.py'
```

If the user already described the change, use that. You don't need to read every model — just enough to write a descriptive message.

### 2. Derive a migration message

Write a concise snake_case message describing the schema change. Keep it under 40 characters (Alembic truncates slugs). Use imperative verb form:

- `add_price_history_table`
- `add_market_cap_to_businesses`
- `drop_legacy_sector_column`
- `rename_ticker_to_symbol`

If the user provided a message, use theirs (converting to snake_case if needed).

### 3. Generate the migration

```bash
just db-migrate <your_message>
```

**If this fails:**
- Connection error → suggest `just db-start` to start dev postgres
- "Can't locate revision" → DB may need initialising, try `just db-upgrade` first
- Import errors → check model imports are correct

### 4. Validate the generated file

Find the newest file in `backend/alembic/versions/` and read it. Check that the `upgrade()` function reflects exactly what you expected from the model diff. Specifically look for:

- **Empty upgrade/downgrade** — autogenerate can't detect server defaults, check constraints, or certain index changes. If the body is just `pass`, the migration is incomplete; explain what's missing and how to add it manually.
- **Unexpected drops** — a column or table being dropped that wasn't intentionally removed is a red flag. Flag it clearly.
- **Missing indexes** — if the model has `index=True` or `unique=True` and the migration doesn't create them, point it out.

If the migration looks correct, say so and apply it:

```bash
just db-upgrade
```

If something looks wrong, explain the issue clearly and stop — let the user decide how to fix it before applying.
